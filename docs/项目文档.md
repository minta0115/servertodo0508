# 待办事项智能提取系统 - 项目文档

## 1. 项目概述

### 1.1 背景与目标

用户工作繁忙，需要从日常沟通（微信、飞书、转发内容）中自动识别和记录待办事项。核心需求：

- **手动输入 + AI 解析**：用户输入一段文字，AI 自动识别其中的待办事项
- **隐式待办识别**：不只是识别"帮我做X"，还要识别"还有件事情没完成"、"X应该要做了"等间接受办表述
- **移动端访问**：手机不在身边时也能访问，电脑可以关机
- **模型切换**：支持切换不同 AI 模型（NVIDIA / MiniMax）

### 1.2 最终成果

- 桌面端和移动端均可用
- 支持截图 OCR 识别待办（前端已集成 Tesseract.js）
- AI 模型可在设置中切换
- 数据存储在云端 PostgreSQL

## 3. 核心功能实现

### 3.1 用户认证

**注册流程**：
```
客户端 POST /api/auth/register
       { email, password, name }
     → 服务器 bcrypt 加密密码
     → 存储到 PostgreSQL users 表
     → 返回 JWT Token (30天有效期)
```

**登录流程**：
```
客户端 POST /api/auth/login
       { email, password }
     → 服务器验证密码
     → 返回 JWT Token
     → 客户端存储到 localStorage
```

**请求拦截**：所有 `/api/*` 接口通过 `authMiddleware` 验证 Token。

### 3.2 AI 待办解析

**核心逻辑** (`server-cloud/src/services/aiParser.js`)：

```javascript
// 根据 API Key 前缀自动识别提供商
function detectProvider(apiKey) {
  if (apiKey.startsWith('nvapi-')) return 'nvidia';
  if (apiKey.startsWith('sk-')) return 'kimi';
  if (apiKey.startsWith('mm-')) return 'minimax';
  return 'openai';
}

// 模型映射
const MODEL_MAP = {
  nvidia: 'meta/llama-3.1-70b-instruct',
  minimax: 'MiniMax-Text-01',
  kimi: 'moonshot-v1-8k',
  openai: 'gpt-4o-mini'
};
```

**解析流程**：
1. 客户端发送原始文本到 `/api/todos/parse`
2. 服务器查询用户设置的 `preferred_provider`
3. 使用用户的 API Key 或系统默认 Key 调用 AI
4. AI 返回结构化待办列表（内容、置信度、截止日期、分类）
5. 逐条存储到 PostgreSQL

### 3.3 模型切换

**前端** (`client/src/components/Settings.jsx`)：
- 展示 NVIDIA / MiniMax 两个选项
- 点击选择，调用 `PUT /api/settings` 保存

**后端** (`server-cloud/src/server.js`)：
```javascript
// /api/todos/parse 中根据 provider 选择 Key
let apiKey = userApiKey;
if (!apiKey) {
  apiKey = preferredProvider === 'minimax' ? DEFAULT_MINIMAX_KEY : DEFAULT_NVIDIA_KEY;
}
```

### 3.4 前端环境变量配置

**开发环境** (`client/.env`)：


**生产环境** (`client/.env.production`)：


**Vite 构建时**：`import.meta.env.VITE_API_BASE` 读取环境变量。

## 4. 部署过程

### 4.1 后端部署到 Railway

1. **创建 Railway 项目**：选择 Empty Project
2. **添加 PostgreSQL**：Railway 自动创建数据库，提供连接字符串
3. **部署 Node.js 应用**：
   - 连接 GitHub 仓库 `E:\AIcode\0011\server-cloud`
   - 设置构建命令 `npm install && npm start`（或 `npm run start`）
   - 设置环境变量（见 4.3）

4. **Railway 环境变量**：

| 变量名 | 值 | 说明 |
|--------|-----|------|
| `DATABASE_URL` | `postgresql://postgres:...@shinkansen.proxy.rlwy.net:29756/railway` | PostgreSQL 连接 |
| `JWT_SECRET` | `todo-secret-2024-railway` | JWT 签名密钥 |
| `PORT` | `3001` | 监听端口 |
| `NODE_ENV` | `production` | 生产模式 |

| `DEFAULT_MINIMAX_API_KEY` | `sk-cp-...` | 系统默认 MiniMax Key |

5. **等待部署完成**：Railway 会返回域名，如 `todo-server-cloud0506-production.up.railway.app`

### 4.2 前端部署到 Netlify

1. **本地构建**：

   - 拖拽 `dist/` 文件夹到 Netlify Dashboard

## 5. 遇到的问题与解决

### 5.1 Kimi API Key 余额不足

**问题**：用户的 Kimi 账户没有余额，API 调用失败。

**解决**：切换到系统默认的 NVIDIA NIM API（免费额度充足）和 MiniMax API。

### 5.2 PostgreSQL 连接超时

**问题**：Railway 部署初期数据库连接超时。

**解决**：等待 Railway 数据库完全初始化，通常需要 1-2 分钟。

### 5.3 浏览器缓存导致 API URL 不更新

**问题**：重新部署后浏览器仍调用旧的 API 地址 `todoservercloud.netlify.app/api`。

**排查过程**：
1. 确认 `.env.production` 和 `vite.config.js` 配置正确
2. `curl` 验证 Netlify 托管的 JS 文件确实包含正确的 Railway URL
3. 确认是浏览器缓存问题

**解决**：
```bash
# 强制刷新
Ctrl + Shift + R          # Windows
Cmd + Shift + R           # Mac

# 或使用无痕模式
Chrome: Ctrl + Shift + N
Safari: Cmd + Shift + N
```

### 5.4 生产环境变量加载时机

**问题**：Vite 中 `import.meta.env.VITE_*` 变量在构建时替换。

**原则**：
- 所有以 `VITE_` 开头的环境变量必须写在 `.env.production` 文件中
- 不能在运行时动态修改
- 构建后的 JS 文件包含的是具体的 URL 值

## 7. 经验总结

### 7.1 环境变量管理

- **敏感信息**：API Keys 只放在后端 `.env`，不提交到 Git
- **前端分离**：前端只配置 API Base URL，不包含任何密钥
- **构建时替换**：Vite 环境下变量在构建时注入，无法运行时更改

### 7.2 前后端分离部署

| 平台 | 优势 | 适用场景 |
|------|------|----------|
| Netlify | 免费、CDN 加速、自动 HTTPS | 静态 SPA |
| Railway | 支持 Node.js + 数据库、一键部署 | 有状态的后端服务 |

### 7.3 AI Provider 选择

- **NVIDIA NIM**：免费额度充足，模型 `llama-3.1-70b-instruct` 效果好
- **MiniMax**：支持 `mm-` 前缀 API Key，响应速度快
- **Kimi（月之暗面）**：需要充值余额，免费额度已用尽

### 7.4 移动端 PWA 支持

React + Vite 构建的项目天然支持 PWA，只需：
```bash
npm install vite-plugin-pwa
```

并在 `vite.config.js` 中配置即可添加 Service Worker 和 manifest。
